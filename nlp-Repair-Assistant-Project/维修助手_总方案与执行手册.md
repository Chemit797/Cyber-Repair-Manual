## 1. 项目定位与评分对齐

**一句话定义**：面向普通用户的nlp-Repair-Assistant——输入自然语言症状，经"症状理解 → 交互式排查 → 文档检索匹配 → 步骤化方案生成"，输出结构化维修指导，超出 DIY 能力时给出送修判断。

| 维度 | 设定 |
|---|---|
| 用户群体 | 非专业维修人员 |
| 设备范围 | 手电筒、电水壶、咖啡机、电风扇（4 类，可扩展） |
| 输入 | 自然语言症状（"按了没反应"）+ 可选设备信息（品牌/型号/大类） |
| 输出 | Step 1/2/3 结构化步骤 + 工具 + 预计耗时 + 安全提示 + DIY/送修判定 |

**rubric 对齐（每分靠什么拿）：**

| 评分项 | 权重 | 我们靠什么 |
|---|---|---|
| Chapter 1 动机/问题 | 5 | 真实痛点（信息不对称导致维修被宰）+ 明确范围 |
| Chapter 2&3 综述+应用复杂度 | **15** | 四子任务全覆盖 + 两阶段检索 + 数据驱动决策树 + 多方法对照（见 §2、§4） |
| Chapter 4&5 结果/分析 | 5 | 检索三方对照 + 分类器 ablation + 混淆矩阵 + 端到端指标（见 §5） |
| Overall 报告质量 | 5 | 结构完整、无语病、≥10 篇文献、IEEE/APA 引用 |
| Presentation & Demo | 10 | 确定性 demo 现场零崩 + 信息密度高的 slides（见 §10） |

**为什么选思路三（五选一结论）**：思路一只实现 4 个子任务里的 1 个、思路二是"智能搜索"非"排查助手"——都会被打到 simple 档。思路三是**唯一四子任务全有真实落地、且不训练、不调 API** 的方案，两天工期下鲁棒性最高。**思路四不丢弃，降级为 Chapter 4 的 ablation**（见 §5.2）。

> 关键澄清：**"不训练/不调 API" 不会拉低 complexity 分。** complexity 评的是 NLP 应用本身的复杂度，不是有没有 fine-tune。两阶段检索 + 置信度门控 + TextRank 抽取式生成 + 多检索器对照，这套已稳在 complex 档；预训练句向量是现代 NLP 标准操作。

## 2. 系统架构（相位划分）

评分人对架构图的第一个问题永远是"**哪些是预先算好的、哪些是实时的**"。所以明确分两相：

**离线相（预先编译）**
- 把语料编译成两个产物：① 检索索引（条目文本 + 句向量），② （可选）训练式分类器（仅用于评测）。
- 决策树 JSON、词典、模板都属于离线静态资源。

**在线相（用户查询时实时跑）**
```
用户症状文本
  → 1. 理解（词典槽位抽取：device + 症状词 + 已做动作 + 指示灯状态）
  → 2. 检索（device 硬过滤 → BM25 召回 top-N → 句向量重排 top-k）
  → 3. 置信度门控（top-1 分数 + top1/top2 margin）
        ├─ 高 → 直接进生成
        └─ 低 → 决策树追问（≤3 轮，问题取自候选条目）→ 重排
  → 4. 生成（模板填充；原文无分点时 TextRank 抽取重组）
  → 结构化输出（步骤 + 工具 + 安全 + DIY/送修判定）
低分兜底：检索分全低 → 诚实告知"未找到精确匹配" + 决策树兜底分支
```

**设计哲学**：在线相**全程确定性、零 LLM、零网络依赖** → 答辩现场不可能崩。这是 demo 拿满分的底层保证。

## 3. 数据契约：故障知识库（阶段 0，最先冻结）

> **schema 一旦冻结就不许改**——它一动所有人返工。这是整个工程的第一优先级。

### 3.1 拆成两份

- **知识库本体（答案侧）**：每条故障的 `possible_causes / solution_steps / tools / safety / ask_questions`，是检索与生成的内容来源。
- **标注症状集（输入侧）**：`symptom_text → fault_category`，每类 ≥10–15 条口语化 paraphrase，供检索测试和（评测用）分类器训练。

### 3.2 单条 schema（JSONL，一行一条）

```json
{
  "id": "fan_hum_no_spin_001",
  "device": "电风扇",
  "fault_category": "嗡嗡响但不转",
  "symptom_text": ["插电后嗡嗡响但扇叶不转", "通电有声音但不转", "电机响但叶片不动"],
  "severity": "需送修",
  "possible_causes": ["启动电容老化", "电机轴卡死/缺润滑", "扇叶被异物卡住"],
  "ask_questions": [
    {"q": "断电后用手拨动扇叶，能顺畅转动吗？", "yes": "电容嫌疑", "no": "轴卡死/异物"}
  ],
  "solution_steps": ["断电拔插头", "清理扇叶周围异物", "手动拨叶检查是否卡死",
                     "若拨动顺畅则多为启动电容老化，需拆机更换电容，建议送修"],
  "tools": ["十字螺丝刀"],
  "est_time_min": 15,
  "safety_notes": ["操作前务必断电拔插头", "电容可能残留余电，非专业人员勿直接触碰"]
}
```

字段约束：`device ∈ {手电筒, 电水壶, 咖啡机, 电风扇}`；`severity ∈ {DIY可修, 需送修}`；`symptom_text` 是字符串数组（多 paraphrase）；其余必填非空。

### 3.3 故障分类草案（taxonomy，可直接用）

| 设备 | fault_category 草案 |
|---|---|
| 手电筒 | 完全不亮 / 亮度暗或忽明忽暗 / 开关失灵 / 间歇性接触不良 / 充电充不进 |
| 电水壶 | 不通电不加热 / 加热但过早自动断电 / 漏水 / 指示灯与加热不一致 / 干烧保护异常 / 水垢异味 |
| 咖啡机 | 不通电不出水 / 出水慢或堵塞 / 漏水 / 咖啡不够热 / 泵异响 / 需除垢 |
| 电风扇 | 不转不通电 / 转速弱无力 / 异响 / 摇头不工作 / 嗡嗡响但不转 / 档位失灵 / 抖动不稳 |

合计约 24 类。目标 **每类 ~12 条 paraphrase → ~300 条**，2 人两三天可完成。

### 3.4 测试集构造（防泄漏，§5 要用）

- 另写 **50–100 条训练里完全没出现过**的 paraphrase，每条标 `gold_fault_category` + `gold_entry_id`。
- 必须由**没参与写对应训练条目**的人来写，避免措辞泄漏。
- 存为独立 `test_set.jsonl`，冲刺期间不许看、不许调参用它（只在 §5 评测时跑一次）。

### 3.5 Claude Code 委派

- 写 `pydantic` schema 校验器 + JSONL 加载器：加载时校验必填/枚举，输出"每个 device × fault_category 的条目数"统计表，**一眼看出哪类样本不足**。
- 写词典骨架文件 `device_dict.txt` / `symptom_dict.txt`（内容人工补，结构 Claude Code 搭）。
- **语料内容人工写**；Claude Code 只做工具链。

## 4. 流水线各阶段（目标 / 算法 / 操作 / Claude Code / 验收）

### 阶段 1 — 症状理解

- **目标**：从自由文本抽出 `device`、症状关键词、`action_taken`（换没换电池）、`indicator_state`（灯闪/不亮/常亮），构造检索 query 并决定进哪棵决策树。
- **算法（baseline，采用）**：jieba 自定义词典匹配 `device`/`component`（闭集，可靠）；正则/模式匹配 `action_taken`、`indicator_state`。
- **Upgrade（不进 demo，仅评测）**：TF-IDF+LR 或 BERT 分类器把症状直接分到 fault_category —— 见 §5.2，作为 ablation。
- **操作**：建词典 → 写抽取函数 → 输出标准化 `Query` 对象。
- **Claude Code 委派**：实现 `SymptomParser` 类（输入字符串，输出 `{device, symptoms[], action_taken, indicator_state}`），含 jieba 自定义词典加载。
- **验收**：手敲 20 句测试句，device 命中率 ≥ 95%，症状词召回肉眼可接受。

### 阶段 2 — 检索匹配（系统脊柱）

- **目标**：把 query 对知识库条目排序，取最贴切的"故障现象–原因–解决步骤"。
- **算法（两阶段级联）**：
  1. `device` 硬过滤 → 候选集只剩该设备条目。
  2. **BM25 召回** top-N（N≈10）：`rank_bm25`，jieba 分词。
  3. **句向量重排** top-k：本地 `BAAI/bge-small-zh-v1.5`（或 `text2vec-base-chinese`），对 N 条算 cosine 重排。处理"嗡嗡响但不转" vs "电机异响伴随停转"这种字面不同意思相同的情况。
  - 知识库仅几百条，**numpy 暴力 cosine 即可，不用 FAISS**（报告写"该规模下暴力检索已足够"反而显得懂权衡）。
- **关键**：句向量模型**本地加载、不训练、不调 API**，启动时一次性编码所有条目并缓存。
- **Claude Code 委派**：实现 `Retriever` 类——`search(query, device) -> [(entry, score)]`，内部 device 过滤 → BM25 → 句向量重排；条目向量缓存到本地 `.npy`。
- **验收**：测试集上级联 P@1 明显高于 BM25-only（§5 出数）。

### 阶段 3 — 交互式排查（决策树，降级为 fallback）

- **目标**：检索不自信时，用最少追问缩小故障范围。
- **触发条件（三信号任一）**：top-1 分数 < 阈值；top1/top2 margin 小；（若启用分类器）熵高。
- **算法**：每设备一棵 JSON 决策树（起点由阶段 1 的 device 决定）；从候选条目取出能区分它们的 `ask_questions`，**一次问一个**，把回答映射成槽位 → 重新过滤/重排；**最多 3 轮**，仍不确定 → 输出"建议送修"。
- **要点**：决策树**数据驱动**（问题来自知识库条目），不是手写大树——报告里这点要写清楚，是区别于"if-else 玩具"的关键。
- **Claude Code 委派**：实现 `DecisionGate`（判断是否追问 + 选下一个区分性问题）和 `DecisionTree` JSON 加载/遍历；写一棵示例树，其余人工填。
- **验收**：低置信场景能在 ≤3 轮收敛到正确叶节点。

### 阶段 4 — 方案生成与呈现

- **目标**：把命中条目渲染成清晰步骤式输出。
- **算法**：
  - **模板填充（主）**：条目本就结构化 → 直接套 `Step 1/2/3 + tools + est_time + safety + 判定`。**确定性、零幻觉。**
  - **TextRank 兜底**：原文是整段无分点 → `jieba.analyse` / `networkx` 做 TextRank 抽 2–3 关键句重组成步骤。
  - **送修判定**：`severity == 需送修` 或检索全低分 → 输出"超出 DIY 能力，建议送修/更换"。
- **Claude Code 委派**：实现 `Renderer.render(entry, confidence) -> str`，含 TextRank 分支与送修判定。
- **验收**：每个 fault_category 至少一条能输出可读的结构化步骤。

### 容错设计（贯穿）

- 检索分全低于阈值 → 诚实告知"未找到精确匹配"，不输出"错误却自信"的答案。
- 决策树设兜底分支，应对数据不全。
- 所有安全相关步骤（断电、触电风险）必须显式提示。

## 5. 评测方案（Chapter 4 弹药）

> 目标：把"complexity"和"analysis"两项分坐实。所有数字/图**落盘成 csv + png**，别只截图。

### 5.1 检索三方对照（核心图）

测试集上对 **BM25-only / dense-only / 级联** 三种配置算 **P@1、P@3、MRR**，出对照表 + 柱状图。预期级联最优——这是报告主结论。

### 5.2 思路四 ablation（把被否的方案变成加分）

TF-IDF + LogisticRegression 训练只要几分钟、零风险。**不进 demo 主链路**，只作离线对照：跑 accuracy / macro-F1 + **混淆矩阵**（看哪些故障互相混，是绝佳 discussion 素材）。报告结论写："我们也试了训练式分类器，发现在本语料规模下，词典+语义检索更稳定"——**一个带分析的负结果比多堆一个模块更值钱。**

### 5.3 端到端 + 消融

- 端到端：诊断正确率、平均追问轮数、送修路由准确率。
- 消融：去掉 device 硬过滤 / 去掉句向量重排，各掉多少分。

### 5.4 Claude Code 委派

- 写 `evaluate.py`：读 `test_set.jsonl`，跑三种检索配置出 P@1/P@3/MRR → csv；matplotlib 画对照柱状图 → png。
- 写分类器评测：sklearn TF-IDF+LR，5-fold CV，混淆矩阵 → png。
- 所有图统一风格、可直接贴进报告。

## 8. Claude Code 使用策略（核心提效手段）

**分工原则**：**人负责判断与契约，Claude Code 负责实现。**

- 人定：schema、taxonomy、阈值取舍、语料内容、报告。
- Claude Code 写：所有类/脚本/UI/评测/画图/环境。

**工作流纪律**：小步交付（一次一个类）、每步先让它跑通再下一步、每个可跑节点 `git commit`、让它写最小单测自检。

**高质量 prompt 模板（可直接改用）**：
1. "用 pydantic 给这个 JSONL schema 写校验器和加载器，校验 device/severity 枚举与必填字段，加载后打印每个 device×fault_category 的条目数。给最小 pytest。"
2. "实现 `Retriever` 类：`search(query, device, topk=3)`。流程：按 device 过滤候选 → rank_bm25 取 top-10 → 用已加载的 sentence-transformer 对这 10 条算 cosine 重排 → 返回 top-k 及分数。条目向量缓存到 .npy，第二次加载直接读缓存。"
3. "实现置信度门控：给定重排结果，若 top1 分数 < τ 或 top1−top2 < δ 则返回 need_clarify=True 及候选 entry 列表，否则返回选中 entry。阈值做成可配置。"
4. "写 `evaluate.py`：读 test_set.jsonl（gold_category, gold_entry_id），对 bm25_only / dense_only / cascade 三配置算 P@1/P@3/MRR，输出 results.csv 并用 matplotlib 画分组柱状图存 png。"
5. "搭 Streamlit：输入框 → 调 pipeline → 渲染 Step 列表 + tools + safety + 判定；need_clarify 时显示追问按钮，点击把答案回传 pipeline 重排。"
